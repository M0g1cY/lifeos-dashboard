# 技术决策记录

## 001 - 第一阶段使用 Mock 数据

**日期**：2026-05-21

**决策**：第一阶段所有数据使用 `src/data/mockLifeData.ts` 管理，不接入数据库。

**原因**：
- 先验证 UI 和信息架构，避免过早接入数据库
- Mock 数据可以快速迭代，不受后端限制
- 确保前端逻辑完整后，再考虑数据持久化

**影响**：
- 开发速度更快
- 可以专注于视觉效果和交互体验
- 后续需要重构数据层（但成本可控）

---

## 002 - MVP 不做登录系统

**日期**：2026-05-21

**决策**：第一版不做用户登录/注册系统。

**原因**：
- 当前是个人使用项目，登录、多用户、权限会增加复杂度
- 专注于核心价值：人生仪表盘的视觉呈现和信息架构
- 登录系统可以在 V2.0 再加入

**影响**：
- 开发周期缩短 2-3 周
- 可以更快看到完整效果
- 后续加入登录系统需要重构状态管理

---

## 003 - 首页优先于功能页

**日期**：2026-05-21

**决策**：Phase 2-3 先把 Dashboard 首页做到精品，再扩展其他页面。

**原因**：
- Dashboard 是 LifeOS 的核心展示面和个人作品集亮点
- 如果首页完成度高，项目马上就有展示价值
- 避免"功能很多但不稳定"的半成品

**影响**：
- 第一版只做 1 个页面（Dashboard）
- 其他页面（/vision、/goals、/habits）放到 V2.0
- 可以更快完成一个高质量的作品

---

## 004 - 航海隐喻是核心设计语言

**日期**：2026-05-21

**决策**：所有功能都必须符合"航海隐喻"的设计语言。

**原因**：
- 航海隐喻是 LifeOS 的核心差异化特征
- 避免变成普通的 Notion/Todo/习惯打卡系统
- 强化品牌记忆点

**影响**：
- 任何新功能都要回答：它是否强化"船长 / 航行 / 北极星 / 里程碑"的隐喻？
- 拒绝不符合隐喻的功能
- 保持产品的独特性和一致性

---

## 005 - 先静态，后动态

**日期**：2026-05-21

**决策**：第一阶段专注静态精品感，不做复杂动画。

**原因**：
- 复杂动画容易拖慢开发
- 静态精品感更容易实现和维护
- 动画可以在 V2.0 逐步增强

**影响**：
- 可以做：圆形进度环、卡片 hover、星空轻微漂浮
- 不要做：复杂 3D 船体动画、大量粒子系统、真实天气 API
---

## 006 - 使用 Next.js 16 而非 Next.js 14

**日期**：2026-05-21

**决策**：继续使用 Next.js 16.2.6，不降级到 Next.js 14。

**原因**：
- `create-next-app@latest` 默认安装了 Next.js 16
- 项目已经完成并通过所有质量检查
- 降级需要重新初始化项目，成本较高
- Next.js 16 是官方最新稳定版本

**影响**：
- 需要适应 Next.js 16 的 API 变化（如 `next lint` 被移除）
- 可能遇到未知的兼容性问题
- 需要更多依赖官方文档和错误提示
- 享受最新的性能优化和特性（Turbopack）

**风险缓解**：
- 遇到问题时优先查阅 Next.js 16 官方文档
- 使用 WebSearch 工具查找最新信息
- 保持小步迭代，及时验证
- 在 `web/AGENTS.md` 中添加提醒：阅读 `node_modules/next/dist/docs/` 中的迁移指南

**已知变化**：
- `next lint` 命令被移除，改用 `eslint .`
- Tailwind CSS 升级到 v4，配置方式改变
- React 19 作为默认版本
- Turbopack 作为默认构建工具

**验证结果**：
- ✅ ESLint 检查通过（0 errors, 0 warnings）
- ✅ 构建成功（4.7s 编译，2.1s TypeScript 检查）
- ✅ 9 个静态路由全部预渲染成功
- ✅ 视觉还原度 92%
